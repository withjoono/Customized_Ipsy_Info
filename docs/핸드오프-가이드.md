## 0. 한 줄 정의
사용자의 **성적·생기부·상황**(= Hub 입시 프로파일)에 맞춰 **최신 입시정보를 골라 배달(push)** 하는 앱. 고1~고3 전체(2015·2022 교육과정).
**검색앱과의 관계 (핵심)**
- **공유**: 인증(Hub SSO), **개인화 레이어(Hub "입시 프로파일" API)**, 기술스택, 공유 DB, 배포 방식, 모든 생태계 컨벤션.
- **다름**: *콘텐츠 엔진*. 검색앱=내부 전형 DB 검색(Pull). 배달앱=**외부 최신 입시정보 수집·큐레이션 + 알림 발송(Push)**.
```
검색앱 엔진 ──┐                              ┌── 배달앱 엔진(이 앱)
 (전형 DB 색인) │                              │   (외부 입시정보 수집 + 푸시)
              └──► [공유: Hub 입시 프로파일 API] ◄──┘
                   생기부+성적+4앱 분석 통합 (Hub가 한 곳에서 계산)
```

## 1. 공통 설정 (검색앱과 동일 — 그대로 적용)
### 1.1 기술 스택 (생태계 표준)
| 영역 | 값 |
|---|---|
| 프론트엔드 | React 18 + Vite + TanStack Router + Zustand + TanStack Query (**npm**) |
| 백엔드 | NestJS 10 + Prisma (**yarn**) |
| DB | 공유 Cloud SQL PostgreSQL (`geobukschool_prod`/`_dev`), **스키마 격리** |
| AI | Vertex AI (Gemini + text-embedding) |
| 배포 | 백엔드 Google App Engine(`ts-back-nest-479305`) / 프론트 Firebase Hosting(`ts-front-479305`) |
### 1.2 데이터 원칙
- 본 앱은 **자기 전용 스키마**만 소유·마이그레이션. 다른 앱 스키마는 절대 쓰기 금지.
- `DB_SYNCHRONIZE` 개념 주의(공유 DB) — Prisma migrate는 자기 스키마만.
- 외부 스키마 읽기는 읽기 전용 계정 + `$queryRaw` 격리(검색앱 `데이터연동-multiSchema-계획.md` §3.2 패턴 A와 동일).
### 1.3 인증 (Hub SSO) — 검색앱과 100% 동일
- Hub가 JWT 발급(HS512, **Base64 secret** `Buffer.from(secret,'base64')`).
- 위성앱 **토큰교환** 방식: client→Hub `login?redirect=`→`sso_code`(Redis 5분)→client backend가 Hub `/auth/sso/verify-code` 호출.
- `AUTH_JWT_SECRET` 동일 값 주입(절대 변경 금지). 서버-서버는 `AUTH_SERVICE_SECRET` + `X-Service-Id`.
- 상세: 검색앱 `Hub-SSO-연동.md` 그대로 참고(앱 이름만 교체).
### 1.4 배포 가드레일 (사고 방지)
- Firebase 배포는 **항상 `firebase deploy --only hosting:<자기-target>`** (target 생략 금지 — 다른 앱 site 덮어쓰기 사고 이력 있음).
- `.firebaserc`엔 자기 site만, `firebase.json`엔 자기 hosting 블록 하나만.

## 3. 데이터 아키텍처 (두 갈래)
### 3.1 개인화 입력 — Hub 입시 프로파일 API (검색앱과 동일하게 소비)
`GET /api/internal/users/:hubUserId/admission-profile` (ServiceAuth, `X-Service-Id: infocast`)
→ 응답: `curriculum(2015/2022)`, `grade`, `naesin_summary`, `csat_mock_summary`, `saenggibu_summary`, `susi_fit_signals`, `jungsi_fit_signals`.
- Hub가 생기부(Hub)+생기부분석(saenggiview `sv_ai_evaluation`)+모의분석(mogo `mg_student_scores`)+수시분석(susi `ss_user_*`)+정시분석(jungsi `js_user_*`)을 종합. 배달앱은 **이 API만 소비**(소스 직접 안 읽음).
### 3.2 콘텐츠 — 최신 입시정보 (이 앱의 고유 엔진)
검색앱에 없는 부분. 별도 수집·큐레이션 파이프라인 필요.
**콘텐츠 종류(예)**: 모집요강 변경, 원서접수/전형 일정, 경쟁률 속보, 입시 정책·뉴스, 대학별 설명회, 마감 임박 알림.
**수집 소스(택/혼합)**:
1. **관리자 입력**(Admin UI) — 가장 신뢰도 높음, 초기 권장.
2. **크롤링/RSS** — 대학 입학처, 한국대학교육협의회(대교협), 어디가 등. (robots/이용약관 준수)
3. **생태계 데이터 변화 감지** — 검색앱 `search.srch_admission` 변경(신규 전형/마감일)을 이벤트로 받아 "너에게 맞는 새 전형" 알림 생성.
**파이프라인**:
```
수집 → 정규화(ic_info_item) → 태깅(대상 학년/계열/지역/전형유형/교육과정)
     → (선택) 임베딩(Vertex) → 사용자 매칭(프로파일 vs 태그/시멘틱)
     → 발송 큐 → 채널 발송(카카오/SMS/푸시/이메일) → 로그
```
### 3.3 본 앱 스키마 (`infocast`, prefix `ic_`) — 초안
| 테이블 | 역할 |
|---|---|
| `ic_info_item` | 입시정보 1건(title, body, category, source, url, target tags(jsonb: 학년/계열/지역/전형/curriculum), published_at, deadline_at) |
| `ic_info_embedding` | 정보 임베딩(시멘틱 매칭용, pgvector) |
| `ic_delivery` | 발송 이력(member_id, info_item_id, channel, status, sent_at, matched_score) |
| `ic_subscription` | 사용자 구독/관심 설정(관심 대학·계열·전형, 채널, 빈도, quiet hours) |
| `ic_match_log` | 매칭 배치 이력 |
> 알림 인프라 참고: 정시앱에 `user_notification_settings`(kakao/sms/push/email + quiet_hours), `notification_logs`, `user_notification_types`(type별 enable/threshold) 구조가 이미 있음 → 동일 패턴 차용 권장.

## 4. 핵심 기능
- **개인화 피드/푸시**: 프로파일·구독에 맞는 정보를 관련도순으로 배달.
- **알림 채널**: 카카오/SMS/푸시(FCM)/이메일 + 사용자별 on/off·빈도·방해금지 시간.
- **이벤트 알림**: 원서접수 D-day, 모집요강 변경, 경쟁률 속보, "내게 맞는 새 전형".
- **관리자**: 정보 입력·승인·발송 관리(크롤링 보완).
- **안전**: 입결/합격가능성은 추정·보장 아님 명시. 외부 출처 표기·링크.

## 5. 구현 순서 (검색앱과 동일한 골격)
```
Phase 0  스캐폴드(검색앱과 동일): 모노레포 + NestJS/Prisma(infocast 스키마) + React/Vite + CLAUDE.md
Phase 1  콘텐츠 인입: ic_info_item 수집(우선 관리자 입력) + 정규화/태깅
Phase 2  구독·매칭: ic_subscription + 프로파일 기반 매칭(정형 태그 우선)
Phase 3  발송: 채널 연동(카카오/SMS/푸시/이메일) + Cloud Scheduler 주기 발송 + 로그
Phase 4  시멘틱 고도화: Vertex 임베딩으로 정보-프로파일 시멘틱 매칭
Phase 5  운영: 모니터링, 빈도/방해금지, 2022 교육과정 확장
```

## 6. 착수 체크리스트 (배달앱)
**공유(이미 검색앱이 요청 중 — 함께 처리)**
- [ ] **Hub 입시 프로파일 API** 구현/공유 (두 앱 공용, 검색앱 체크리스트 ②-b와 동일 항목)
- [ ] `AUTH_JWT_SECRET`(Base64) 동일 값 공유, `AUTH_SERVICE_SECRET` 서비스 토큰(`X-Service-Id: infocast`)
**배달앱 전용**
- [ ] Hub `hub_apps`에 `infocast` 등록 + Hub Frontend에 `VITE_INFOCAST_URL` env 추가(sso_code 생성용)
- [ ] DB: `infocast` 스키마 생성 권한 + (생태계 변화 감지 시) `search` 등 읽기 권한 협의
- [ ] GCP: App Engine `infocast-backend` / Firebase site `infocast-front` 생성, 배포 워크플로(다른 위성앱 복제)
- [ ] 알림 채널 키: 카카오 알림톡, SMS(예: 알리고/NHN), FCM, 이메일(SMTP/SendGrid)
- [ ] 콘텐츠 수집 방식 확정(관리자 입력 우선 / 크롤링 소스 목록 / 검색앱 변경 이벤트 연동)

## 7. 함께 정할 결정
1. 앱 이름·service/site·스키마·포트 확정(§2 제안 검토)
2. 콘텐츠 1차 수집 방식: 관리자 입력 vs 크롤링 vs 이벤트 — 우선순위
3. 발송 채널 우선순위(카카오/SMS/푸시/이메일 중 1차)
4. 매칭 기준: 정형 태그 우선 → 시멘틱 추가 시점
5. (선택) "검색앱 전형 변경 → 배달 이벤트"를 위해 검색앱과의 연동(Pub/Sub vs `srch_admission` 폴링)
